需求背景

这里先说一下场景

公司和甲方签的合同 , 做一款程序,并且维护3年后,甲方可获得项目源码
但现在甲方很聪明,不想付维护的钱,但合同里又拥有项目源码的产权
现在甲方’违约’ 不想出维护的费用,但项目的钱已经付了,我们还是要给出源码
因为我们项目中有部分代码是公司自有产权,并不想直接连这部分一起直接给甲方,于是老板就想我们把这部分代码加密后给甲方,不让甲方获得这部分代码
众所周知 java是一款一处编译处处运行的编程语言,编译成class文件后可以让虚拟机识别,但也让反编译工具 十分容易获得其’源码’

主要影响漏洞

fastjson的@type漏洞,攻击者可向目标服务器发起攻击json请求,当目标服务器反序列化该攻击json串时,则会引发反序列化漏洞,实现挂载恶意java类,达到攻击目的

Fastjson反序列化漏洞被利用的原因，可以归结为两方面：

  1. Fastjson提供了反序列化功能，允许用户在输入JSON串时通过“@type”键对应的value指定任意反序列化类名；
  2. Fastjson自定义的反序列化机制会使用反射生成上述指定类的实例化对象，并自动调用该对象的setter方法及部分getter方法。

攻击者可以构造恶意请求，使目标应用的代码执行流程进入这部分特定setter或getter方法，若上述方法中有可被恶意利用的逻辑（也就是通常所指的“Gadget”），则会造成一些严重的安全问题。

Lock和Condition

锁和条件变量,jdk1.5引入的接口,可以帮助我们解决共享资源互斥的问题

Lock：锁

  • lock():获取一个锁
  • unlock()解锁
  • newCondition()创建一个条件变量
  • tryLock()尝试获取锁，获取了锁返回true，否则false

常见实现:如ReentrantLock，即可重入锁，可以理解为是java关键字synchronized,代码层面的实现，其功能是一样的但其底层原理实现是不一样的

Lock底层实现一般都是CAS原语(compareAndSwap)
synchronized底层实现是Monitor(管程，或叫监视器)

Condition：条件变量

一把锁可以有多个条件变量，使用lock.newCondition()来创建Condition

await(): 线程阻塞等待当前条件变量，需要其他线程调用对应实例的signal来唤醒
signal(): 唤醒一个此条件变量等待的线程
signalAll(): 唤醒全部此条件变量等待的线程

使用锁的最佳原则

并发大师 Doug Lea《Java 并发编程：设计原则与模式》一书中，推荐的三个用锁的最佳实践，它们分别是

  1. 永远只在更新对象的成员变量时加锁
  2. 永远只在访问可变的成员变量时加锁
  3. 永远不在调用其他对象的方法时加锁

acquire()，获得一个凭证，当凭证小于等于0时，此线程阻塞
release()，释放一个凭证,permits+1

ReadWriteLock

读写锁

  • 适用于读多写少
  • 允许多个线程获取读锁
  • 只允许一个线程获取写锁
  • 只允许锁的降级，不允许锁的升级
  • 锁的升级:获取读锁时不释放读锁的情况下去获取写锁
  • 锁的降级:获取写锁时不释放写锁的情况下去获取读锁
  • 获取读锁的前提是没有线程在使用写锁
  • 获取写锁的前提是没有线程在使用读锁或写锁
  • 获取写锁不会中断其他线程获取读锁

StampedLock

标记锁，是读写锁的改进,其实现思想类似于乐观锁

  • 支持写锁、悲观读锁和乐观读
  • 其中，写锁、悲观读锁的语义和 ReadWriteLock 的写锁、读锁的语义类似

CountDownLatch 和CyclicBarrier

使用计数器来帮助协同线程工作

  • CountDownLatch ：常用于一个线程等待其他线程
  • CyclicBarrier：常用于几个线程之间互相等待进行同步操作

CountDownLatch

  • 类似于计数器,在减到0之前会被阻塞
  • countDown()，计数器减少
  • await()，线程等待，如果计数器大于0

CyclicBarrier

  • 类似于CountDownLatch,创建时需提供一个count，也可传入一个Runnable,当计数器减到0时调用
  • 当计数器减到0时会自动重置

Queue

Java 并发包里面 Queue 这类并发容器是最复杂的，你可以从以下两个维度来分类。
1.维度是阻塞与非阻.塞，所谓阻塞指的是当队列已满时，入队操作阻塞；当队列已空时，出队操作阻塞。
2.是单端与双端，单端指的是只能队尾入队，队首出队；而双端指的是队首队尾皆可入队出队。
Java 并发包里阻塞队列都用 Blocking 关键字标识，
单端队列使用 Queue 标识，双端队列使用 Deque 标识
另外还有一点需要额外注意，使用队列时千万要注意队列是否为有界队列,如为无界队列,当数据量太大时可能会OOM,而为有界队列时则需要注意拒绝策略

单端阻塞队列
其实现有 ArrayBlockingQueue、LinkedBlockingQueue、SynchronousQueue、LinkedTransferQueue、PriorityBlockingQueue 和 DelayQueue。内部一般会持有一个队列，这个队列可以是数组（其实现是 ArrayBlockingQueue）也可以是链表（其实现是 LinkedBlockingQueue）；甚至还可以不持有队列（其实现是 SynchronousQueue），此时生产者线程的入队操作必须等待消费者线程的出队操作。而 LinkedTransferQueue 融合 LinkedBlockingQueue 和 SynchronousQueue 的功能，性能比 LinkedBlockingQueue 更好；PriorityBlockingQueue 支持按照优先级出队；DelayQueue 支持延时出队。

Executor

线程池，代表类:ThreadPoolExecutor
可使用静态工厂方法来创建
Executors.newFixedThreadPool()…等
但在各个大厂的开发规范里已经明确规定了严禁使用此静态工厂方法论来创建线程池
必须直接使用new ThreadPoolExecutor()方法来创建
最主要的原因是，默认使用静态方法来创建都是无界队列，而这在生产环境如任务积累过多则可能导致OOM!所以一定要注意
创建线程池最好使用有界队列!

ThreadPoolExecutor参数列表详解

  • corePoolSize：表示线程池保有的最小线程数。有些项目很闲，但是也不能把人都撤了，至少要留 corePoolSize 个人坚守阵地。
  • maximumPoolSize：表示线程池创建的最大线程数。当项目很忙时，就需要加人，但是也不能无限制地加，最多就加到 maximumPoolSize 个人。当项目闲下来时，就要撤人了，最多能撤到 corePoolSize 个人。
  • keepAliveTime & unit：上面提到项目根据忙闲来增减人员，那在编程世界里，如何定义忙和闲呢？很简单，一个线程如果在一段时间内，都没有执行任务，说明很闲，keepAliveTime 和 unit 就是用来定义这个“一段时间”的参数。也就是说，如果一个线程空闲了keepAliveTime & unit这么久，而且线程池的线程数大于 corePoolSize ，那么这个空闲的线程就要被回收了。
  • workQueue：工作队列，和上面示例代码的工作队列同义。threadFactory：通过这个参数你可以自定义如何创建线程，例如你可以给线程指定一个有意义的名字。
  • handler：通过这个参数你可以自定义任务的拒绝策略。如果线程池中所有的线程都在忙碌，并且工作队列也满了（前提是工作队列是有界队列），那么此时提交任务，线程池就会拒绝接收。至于拒绝的策略，你可以通过 handler 这个参数来指定。
    ThreadPoolExecutor 已经提供了以下 4 种策略。
    CallerRunsPolicy：提交任务的线程自己去执行该任务。
    AbortPolicy：默认的拒绝策略，会 throws RejectedExecutionException。
    DiscardPolicy：直接丢弃任务，没有任何异常抛出。
    DiscardOldestPolicy：丢弃最老的任务，其实就是把最早进入工作队列的任务丢弃，然后把新任务加入到工作队列。

注意事项:

throw RejectedExecutionException

Fork/Join：单机版的MapReduce

关键字:

核心思想为分治,即分而治之:将任务拆分至一个个子任务至不可拆分,后将任务结果逐个合并至最终结果
知识点总结
支持任务窃取,保证不会有的线程程很忙，而有的线程很闲的状况

如何优雅地终止线程池

线程池提供了两个终止方法：**shutdown()和shutdownNow()**。

调用这两个方法后线程池都会拒绝接受新任务

shutdown() 方法是一种很保守的关闭线程池的方法。
线程池执行 shutdown() 后，就会拒绝接收新的任务，但是会等待线程池中正在执行的任务和已经进入阻塞队列的任务都执行完之后才最终关闭线程池。

shutdownNow() 方法，相对就激进一些了，
线程池执行 shutdownNow() 后，会拒绝接收新的任务，同时还会中断线程池中正在执行的任务，已经进入阻塞队列的任务也被剥夺了执行的机会，不过这些被剥夺执行机会的任务会作为 shutdownNow() 方法的返回值返回。因为 shutdownNow() 方法会中断正在执行的线程，所以提交到线程池的任务，如果需要优雅地结束，就需要正确地处理线程中断。

执行流程

  1. 解析sql,获取片键值
  2. sJDBC 通过配置的库策略,表,来判断sql往哪个库,哪个表来发出请求操作
  3. sJDBC 改写逻辑表名sql,为真正要执行的sql
  4. sJDBC执行真正的sql
  5. sJDBC汇总sql执行结果

查询

查询中,如果有 in,id中包含多个不同的库,则sjdbc会将sql拆分为访问不同库或表的sql,再将结果组合到一起

绑定表

指分片键完全相同,则此两张表互为绑定关系

如非绑定表关系,则在join查询时,会出现笛卡尔积关联

如 t_order 和 t_order_item , t_order_item 和 t_order 是多对一的关系

广播表

指所有分片数据源中都存在的表,表结构和表中数据在每个数据库中均完全一致,适用于数据量不大,且需要与海量数据关联查询的场景,如字典表

SQL执行,连接模式

sJDBC采用一套自动化的执行引擎,负责将路由和改写完的真实sql安全且高效发送到底层数据源执行。

它不是简单地将SQL通过jdbc直接发送至数据源执行,也并非直接将执行请求放入线程池去并发执行。

它更关注平衡数据源连接创建及内存占用所产生的消耗,以及最大限度地利合理利用并发等问题。

执行引擎的目标是自动化的平衡资源控制与执行效率,他能在以下两种模式自适应切换

内存限制模式

sJDBC,对一次操作所耗费的数据库连接不做限制。

如果实际执行的sql需要对某数据库实例中的200张表做操作,则对每张表创建一个新的数据库连接,并通过多线程的方式并发处理,追求执行效率的最大化

连接限制模式

严格限制sJDBC所耗费的数据库连接数量,如实际执行的sql需要对某数据库实例中的200张表做操作,那么只会创建唯一的数据库连接并对200张表进行串行处理

如果一次操作中的分片散落在不同的数据库中,仍然采用多线程对不同库的操作,但每个库的每次操作仍然只创建一个数据库连接

内存限制模式适用于OLAP应用,连接限制模式适用于OLTP应用

浅谈高并发下的,超卖问题

常见易错问题

更新库存时常用sql如下，但初学者往往会犯下如下错误

问题1:脏数据

在java代码中计算出库存后再set值
此方法不可取,很容易会发生并发问题,导致覆盖了库存数量

问题2:原子性

使用sql保证其原子性
数据库的排它锁:因为update时会有排它锁,在更新的时,此行会被锁定，只可读不可更新

如在事务中，则在事务提交前时，其他更新操作将会被阻塞
更多update排它锁为何可保证原子性可参考
阿里mysql大佬给出的update并发解释

但仍存在超卖即库存可能为0的问题
同理，我们仍可以如此来让数据库帮我们做库存判断问题

另外需要注意一定要判断update返回的更新行数,如为0则说明未能更新到库存

问题3:死锁

上面的问题理论上在事务管理的情况下已经解决了更新库存的原子更新问题，和库存超卖问题

但是在实际运用中，可能仍存在问题:

在事务管理中，当两个线程同时更新A和B两个商品

  • T1线程先更新了A商品，后更新B商品
  • T2线程先更新了B商品, 后更新A商品

而此时如果正好执行在中间，即线程T1持有商品A的互斥锁，线程T2持有商品B的互斥锁，后再去获取另一个商品的互斥锁，这样因互相等待就形成了死锁
这样的结果往往会导致事务超时，或被Spring检测出死锁，使其中一个线程抛出异常中断，另一个线程获得锁执行
而这样明显是不友好的

解决方案

所以我们可以考虑使用其他的锁，来保证互斥性
能实现锁达到互斥的作用的方法有很多，
如java关键字synchronized，jdk中的Lock等
但在使用保证护持时需要遵循一个原则:
在访问一个共享资源时,锁是唯一的

在高并发场景中一定要保护好瓶颈资源:数据库

单机场景

所以我们应该分场景使用锁
如为单机应用的情况下，操作库存的互斥完全可以使用synchronized,CAS原语,Lock等来实现
这里建议使用Lock或CAS，来给锁增加一个超时来避免线程死锁

分布式场景

但在分布式，集群场景时，使用jdk提供的锁就不能保证互斥了，因为不同的虚拟机互相创建的对象是肯定不一样的，所以也就不能达到一把锁的条件

这时可以使用redis来作为分布式锁

网上实现有很多，基本上都是使用setnx语句来设置，如key存在则表明有被锁，而不存在则表明没有被锁

详情实现可以参考分布式锁用Redis还是Zookeeper

  • 另外高并发场景，如并发量太高的话，可能是需要消息队列来削峰,减缓服务器压力的
  • 如果想进行并发压测，用apache自带的ab或apache JMeter都是可以的
